> 對應架構圖第 ① 格(左上,黑色 placeholder)。
> 學習時間:15 ~ 30 分鐘。
> 必修角色:**全員**(高層 / PM ✅深;其他 ✅淺)。
> 上層銜接:[U2 Agentic Engineer 角色](U2-Agentic-Engineer-Role.md)。

## 一、單元目標

學完這個單元,你應該能回答:

1. 公司為什麼要把 AI / Agent 導入既有系統?**不是為了趕時髦**,而是要解什麼具體痛點?
2. 業務情境如何決定後面 4 層(Agentic Engineering、Harness、Flywheel、基礎資源)的優先順序?
3. 業務情境跟 Coolify Deploy 看似無關,為什麼還是會影響上線節奏?

---

## 二、圖中對應內容

架構圖第 ① 格的核心命題:

```
目標:提升交付效率、知識複用、跨部門協作
```

驅動方向:`業務情境 → 設定目標 → 上層工作流(③) → 執行底座(④) → 基礎資源(⑤)`。

換言之,**所有架構選擇,最終都要可以回答「這對公司業務有什麼幫助」**。如果一個 Agent 工作流無法回答這個問題,就不該被建。

---

## 三、公司目前的三大痛點

對應 [`20260421-DataFlow-v1.0.md`](../20260421-DataFlow-v1.0.md) 第一節:

1. **交付效率低** — 同樣需求,不同團隊各自做兩套相似系統。
2. **知識難複用** — SOP、PLM/BOM、過往專案文件散落,新人 / AI 都讀不到。
3. **跨部門協作斷點多** — 登入各做各的、資料各存各的、部署各做各的。

Agent 化的目標是把這三件事**自動化、可審查、可重用**,而不是疊一層 chatbot。

---

## 四、業務情境如何決定後續層級

| 業務優先 | 對應後續層的優先 |
| --- | --- |
| 「先把 SOP 知識複用起來」 | U5(SOP/規格文件)+ U4(文件/知識/Memory)優先 |
| 「先解決測試 Log Triage 太花時間」 | U3 的「測試/驗證 Agent」+ U4 的「Logs/Traces/Observability」優先 |
| 「先做設計變更摘要」 | U3 的「設計 Review Agent」+ U5 的「Repo/Code」優先 |
| 「未來要做專屬 Domain Model」 | U4.5 飛輪整套必須先鋪設(資料採集 → Evals → Trajectories) |

→ **沒有業務情境驅動的架構建設,等於在堆積技術債。**

---

## 五、與 Coolify Deploy 的關聯

業務情境本身**沒有直接的 Coolify 合規項**,但會影響:

1. **SLA 與部署頻率** — 業務若要求「測試 Log Triage 必須當天回」,就必須要求 Coolify 的部署能在 5 分鐘內完成,不能堆排程。
2. **資料邊界** — 若業務情境涉及客戶資料 / 員工資料,Coolify 上的 Container 必須與資料中心(見 DataFlow v1.0)走內網,不可走公網。
3. **核准鏈** — 業務優先序高的功能,deploy 簽核權通常上收到 Tech Lead;優先序低的可由 Owner 自簽。這條規則寫在 [Coolify-Compliance-Mapping.md](Coolify-Compliance-Mapping.md) 第 6 條(Audit Trail)。

## 六、自我檢核

通過這 5 題即視為 U1 完成:

1. 寫出公司目前最痛的 3 個業務情境,並指出每個對應「現在可落地用途」的哪一項(架構圖右上角 5 項)。
2. 給定一個新 Agent 提案,說明它對應上面哪個業務痛點;如果都不對,該不該做?
3. 解釋為什麼「設計變更摘要」優先級會高於「跨案知識協同」(提示:落地難度 + 資料現況)。
4. 說明業務情境如何影響 Coolify 部署頻率的 SLA。
5. 找出 [`20260421-DataFlow-v1.0.md`](../20260421-DataFlow-v1.0.md) 中,本單元第三節三大痛點分別對應第幾節。
