## 三、6 個飛輪元件詳解

### ① 公司資料 / 任務來源
- **作用**:飛輪的燃料 — 真實業務資料(SOP、PLM、Test Logs、客服對話、設計變更紀錄)。
- **公司來源**:見 [`DataFlow v1.0`](../20260421-DataFlow-v1.0.md)「企業中央資料中心」。
- **關鍵約束**:**資料必須脫敏 + 標註用途**(訓練 / Eval / 兩者),否則卡在合規。

### ② Evals(可驗證任務)
- **作用**:把模糊的「品質好」翻成「答對率 ≥ 80%」「Triage 準確率 ≥ 90%」這類**可量測**指標。
- **與 U4 的關係**:Evals 進到 U4.5 Tests/CI suite,每次 Prompt / Model 改動都跑一次。
- **常見錯誤**:Eval 集太小(< 50 題)→ 數字噪音蓋過訊號。

### ③ Reward / Success Signal
- **作用**:標記哪些 trajectory 是「好」的、哪些是「壞」的。
- **公司來源**:Agentic Engineer(U2)的審查結果是最權威的 reward 來源;次選是業務使用者點讚 / 點爛。
- **關鍵約束**:reward 不可由 LLM 自己評(self-reward bias)。

### ④ Trajectories / Failure Cases
- **作用**:把「Agent 在某個任務上的完整執行路徑(Prompt → Tool 呼叫 → 結果)」存起來。
- **存在哪**:獨立的 trajectory store(可以是 Postgres + 物件儲存,Coolify 起一份)。
- **Failure Cases 用途**:回去調整 Prompt(短期)或進入 Fine-tuning 訓練集(長期)。

### ⑤ Fine-tuning / LoRA / RLVR
- **作用**:把 trajectory + reward 餵給訓練,讓模型「學會」公司的判斷標準。
- **3 種選擇**:
  - **Fine-tuning**:全參數,昂貴,效果最好。
  - **LoRA**:輕量微調,公司資源下的主力選擇。
  - **RLVR**(Reinforcement Learning with Verifiable Rewards):用 reward signal 直接強化,適合「答案可驗證」的任務(例:程式碼測試、SQL 查詢)。
- **Coolify 合規**:訓練本身**不在 Coolify 上跑**(算力不夠),走外部 GCP/AWS Serverless GPU(對應公司 DevOps 限制);訓練結果(LoRA weights)透過 image 帶回 Coolify 部署。

### ⑥ 公司專用 Domain Model
- **作用**:飛輪終點,新一代的 Model,接回 U4 的 `Agent = Model + Harness`。
- **驗收**:必須通過 ② Evals 的全套指標,且優於上一版至少 X%。
- **版本管理**:Model version 寫進 Audit Trail(對應附件第 6 條)。

## 四、為什麼順序不能跳

| 跳過的元件 | 後果 |
| --- | --- |
| 跳過 ① 資料 | 沒燃料,飛輪轉不起來。 |
| 跳過 ② Evals | 沒度量,改了不知道有沒有變好。 |
| 跳過 ③ Reward | 訓練沒方向,模型亂收斂。 |
| 跳過 ④ Trajectories | 只有 Q&A 對,學不到「為什麼這樣判斷」的中間步驟。 |
| 跳過 ⑤ Fine-tuning | 永遠停在 Prompt 工程,模型本身不會進化。 |
| 跳過 ⑥ Domain Model | Flywheel 沒輸出,只是個分析系統。 |

→ **6 個元件是嚴格順序;任何一環斷了,飛輪停轉**。

## 五、什麼時候才該啟動飛輪

**不是每個 Agent 都需要 Domain Model**。建議啟動條件:

1. 已累積 **≥ 6 個月真實任務 trajectory**(< 6 個月資料量不夠)。
2. 通用 LLM 在這個任務上**穩定 plateau**(改 Prompt 改不上去)。
3. 任務有**清楚的 reward signal**(對 / 錯可判定)。
4. 業務願意接受**訓練週期**(LoRA 約 1 ~ 2 週、Fine-tuning 約 1 個月)。

公司目前(2026-05)推薦先把 **U1 ~ U4 落實完整**,U4.5 在第二年再啟動。

## 六、與 Coolify Deploy 的合規對照

| Coolify 合規通則 | 與 U4.5 的關聯 |
| --- | --- |
| 1. 容器與映像 | LoRA weights 帶進 image,但不可同時放多版本(image 暴肥) |
| 2. Secret 管理 | 訓練資料來源(資料中心 token)走 Coolify env,不入訓練 image |
| 3. 權限與隔離 | Trajectory store 必須與業務 DB 隔離(避免 PII 洩漏) |
| 4. 可觀測性 | Eval 結果必須走 Dashboard,可被 Agentic Engineer 看見 |
| 5. CI/CD | Model 版本切換必須走 Coolify deploy 流程,不可線上熱換 |
| 6. Audit Trail | Model 版本 → 訓練資料來源 → 簽核者,完整可回溯(法遵與責任歸屬) |

通過這 8 題即視為 U4.5 完成:

7. Model 版本切換為什麼不可線上熱換?(對照 U4.1 Prompts 的同樣原則)
