# 企業內部資料流整合 v1.0
## 技術藍圖簡報

> 公司所有內部系統的**登入、開發、資料、部署、AI**,五件事整合成**同一條資料流**。

2026-04-21 · IT 部門 · 讀者:IT / 開發者

---

## 為什麼要這份藍圖

公司過去**各系統獨立成長**,沒有一條貫通的資料流:

- **登入各做各的** — 每個系統自建帳號、自建權限
- **開發各做各的** — 同樣需求做兩套不相容版本
- **資料各存各的** — 基礎資料在多處被複製維護
- **部署各做各的** — 每套系統一台 VM,idle 也整月計費
- **AI 做不了** — 資料散亂,RAG 無法有效索引

> 核心口訣:**一次登入、一套規範、一份資料、一個部署平台、一個知識庫**

---

## 整體資料流總覽

```
     Microsoft AD(單一身分來源)
              ↓
     統一登入閘道 (AD 登入器)
              ↓
  ┌─────────────────────────────────────┐
  │  業務核心項目系統(遵循 AI-Spec)   │
  │  ERP / CRM / BPM / 跨部門工具 ...   │
  └──────────────┬──────────────────────┘
                 ↓ 讀寫
  ┌─────────────────────────────────────┐
  │  企業中央資料中心(Data Hub)       │
  │  員工 / 客戶 / 幣別 / 代碼 / 會計    │
  └────────┬─────────────┬──────────────┘
           ↓             ↓
      Seq (log)      AI RAG 知識庫

  所有服務 → Coolify(共用 Docker 基礎設施)
```

## 五大核心模組

| # | 模組 | 對應痛點 |
| --- | --- | --- |
| 1 | 統一登入入口(AD 登入器) | 帳號各自為政 |
| 2 | 開發標準規範 | 系統長得不一樣 |
| 3 | 中央資料中心(Data Hub) | 資料重複維護 |
| 4 | Coolify 部署管理 | 雲端成本失控 |
| 5 | AI RAG 知識庫 | AI 做不起來 |

---

## 模組 1 + 2 — 登入 & 開發規範

**① 統一登入入口**
- 以 **Microsoft AD** 為單一身分來源,自建閘道代理所有驗證
- 協議 OIDC / SAML;Session Redis;單一登出即時生效
- **所有新系統一律整合此閘道,不得自建帳號**

**② 開發標準規範**
- 架構 / API / DB / 部署 / 資安 / Log 的共同底線
- 新專案套規範起骨架,不從零設計
- 既有專案以 `/scan-project` 偵測偏差

> 詳見 `CLAUDE-v1.0.md`、`spec/*`、`sso-init-*`、`scan/*`

---

## 模組 3 — 企業中央資料中心(Data Hub)

**主檔範圍**:組織 / 人員 / 客戶 / 代碼表 / 會計 / 商品

**技術骨幹**
- 儲存:PostgreSQL + Adminer
- 對外:統一的主檔 REST 服務
- 變更推播:CDC / 事件訂閱
- 對接舊系統:ETL(涵蓋既有 SaaS 雙向同步)

**架構規範**
- 業務系統**只讀**主檔,禁止自建重複欄位
- 主檔語意變更必須**升版 + 過渡期**
- 所有寫入走 audit log

---

## 模組 4 + 5 — Coolify 部署 & AI RAG

**④ Coolify 部署管理**
- 所有自建系統的**唯一部署平台**(Docker Compose)
- 負責 build、HTTPS、環境變數、log 集中、回滾
- **多服務共用一台機器**
- 標準服務:Postgres + Adminer + Seq

**⑤ AI RAG 知識庫**
- 資料:Data Hub + 業務摘要 + 文件庫 + Seq log
- 技術:pgvector / Qdrant、依需求選 embedding 與推論模型
- **四大鐵律**:髒資料不索引、PII 脫敏、回答附來源、權限不繞過

---

## 兩大根本問題

### 問題 1 — 各系統重複建資料表
- **現象**:員工、客戶、幣別在各系統各自維護,改一邊忘一邊
- **對策**:Data Hub 為唯一寫入來源;規範禁止重複;`/scan-project` 擴充偵測
- **成功**:新系統主檔類欄位 0 複製;離職 1 小時內全系統失效

### 問題 2 — 雲端機器成本偏高
- **現象**:流量極低的工具獨佔 VM,整月 idle 仍計費
- **對策**:統一上 Coolify 共用機器、資源邊界、半年未用下線
- **成功**:單機承載 ≥ 5 個服務、月帳單遞減

---

## 雲端成本是「架構問題」

> 成本**不是財務議題,是部署模式問題**。
> 省錢 ≠ 砍大機器;**省錢 = 改變部署模式**。

**常駐費用拆解**(新增一個系統就疊加一份,0 流量也照付):

| 項目 | 付費? | 項目 | 付費? |
| --- | --- | --- | --- |
| VM 常駐費 | ✅ | 備份快照 | ✅ |
| 區塊儲存 | ✅ | 監控 agent | ✅ |
| 固定 IP / LB | ✅ | 流量 egress | ❌ |

---

## 部署模式對照

> **公司現況**:無 K8s;Serverless 僅限 GCP/AWS

| 模式 | 成本(低) | 運維 | 我們能用嗎 |
| --- | --- | --- | --- |
| 一系統一 VM(現狀) | 🔴 高 | 🟡 中 | 最貴 |
| Coolify 共用(目標) | 🟢 低 | 🟢 低 | ✅ 主力 |
| Serverless (GCP/AWS) | 🟢 極低 | 🟢 低 | ⚠️ 限特定場景 |
| Replit / 託管 | 🟢 低 | 🟢 極低 | ✅ 個人工具 |
| ~~Kubernetes~~ | — | 🔴 高 | ❌ 無基礎建設 |

**Serverless 適合**:webhook、定時批次、偶發流量、對外靜態
**Serverless 不適合**:有 session / 長連線 / 穩定流量的 web app

## 成本優化與分層

**優化手法(依影響力)**
1. **共用基礎設施** — 多服務一台機、共用 Postgres(schema 隔離)
2. **分層部署** — Tier 1 / 2 / 3 差異化資源配置
3. **資源邊界** — 每 service 設 CPU / memory limit
4. **排程啟停** — 非上班時段關 T2/T3
5. **儲存分層** — log TTL、備份冷存、檔案 IA
6. **CDN / 監控** — 前端走 CDN;Billing 閾值告警

**分層**

| Tier | 範例 | 策略 |
| --- | --- | --- |
| **T1 生產關鍵** | 業務核心系統、財務、對客 | 獨立資源、嚴格 SLA |
| **T2 內部工具** | HR 簽核、報表 | 共用 Coolify + DB schema 隔離 |
| **T3 實驗 / 個人** | Vibe Coding、POC | 最小規格、排程啟停、自動下線 |

> 典型:5 台 VM 合併成 1 台,帳單降 ~70%

## Trade-off 與治理原則

**選擇低成本,要換的是**

| 代價 | 對策 |
| --- | --- |
| 共用機器 = 單點故障放大 | 標準化備份 + 1 小時重拉 |
| 共用 DB = noisy neighbor | 資源邊界、慢查詢監控、T1 獨立 |
| 排程啟停 = dev 體驗差 | 排程外手動喚醒 |
| 壓太緊 = 高峰當機 | 保留 20-30% 緩衝 |
| 下線機制 = 可能誤殺 | 預告期 + owner 確認 |

**治理原則**
- **誰建的誰負責**:上線登記 owner、預期流量、預期成本
- **季度檢視**:top N 高成本服務合理性審視
- **預算上限**:各 Tier 月度總額超過告警並凍結上線

> **T1 絕不為省錢妥協穩定**;T2/T3 積極節流

## 資料流情境實例

**A. 新員工入職**
HR 在 AD 建帳號 → 閘道同步 → Data Hub 寫入主檔 → 員工登入業務系統(AD 驗證 + 讀 Data Hub 取角色) → Seq 記 log → AI RAG 更新索引

**B. 主檔變更**
使用者更新主檔 → 業務系統呼叫 Data Hub API → 寫入 + 發 CDC 事件 → 下游系統 & AI RAG 同步 → Seq 留完整 audit

**C. 新系統上線**
套 template → `sso-init-*` 接閘道 → 接 Data Hub → 寫 compose → Coolify 自動部署 → `/scan-project` 通過 → log 進 Seq、資料進 AI RAG

## 一張投影片總結

- 公司缺一條**統一資料流**,每件事都在各自造輪子
- 解方是**五個統一**:登入、開發、資料、部署、AI
- 兩個根本痛點:**資料重複** + **成本失控**
- 部署走 **Coolify** 共用基礎設施;K8s 不在選項內
- AI 放最後,**前四穩了才值得做**

> 「**整合一次,長期收租**」— 整個藍圖的經濟學
