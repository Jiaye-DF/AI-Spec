# 企業內部資料流整合 v1.0
## 技術藍圖簡報

> 2026-04-21 · IT 部門
> 目標讀者:IT 團隊、開發者、要建置新系統的工程師

---

## 一句話定位

> 公司所有內部系統的**登入、開發、資料、部署、AI**,五件事整合成**同一條資料流**。

讓新系統啟動時知道「每一段」該接到哪裡、用哪套規範、產物放哪。

---

## 為什麼要這份藍圖

公司過去幾年**各系統獨立成長**,沒有一條貫通的資料流:

- **登入各做各的** — 每個系統自建帳號、自建權限
- **開發各做各的** — 同樣需求做出兩套不相容版本
- **資料各存各的** — 基礎資料在多處被複製維護
- **部署各做各的** — 每套系統一台 VM,idle 也是整月計費
- **AI 做不了** — 資料散亂、格式不一,RAG 無法有效索引

---

## 整體資料流總覽

```
     Microsoft AD(單一身分來源)
              ↓
     統一登入閘道 (AD 登入器)
              ↓
  ┌─────────────────────────────────────┐
  │  業務核心項目系統(遵循 AI-Spec)   │
  │  ERP / CRM / BPM / 跨部門工具 ...   │
  └──────────────┬──────────────────────┘
                 ↓ 讀寫
  ┌─────────────────────────────────────┐
  │  企業中央資料中心(Data Hub)       │
  │  員工 / 客戶 / 幣別 / 代碼 / 會計    │
  └────────┬─────────────┬──────────────┘
           ↓             ↓
      Seq (log)      AI RAG 知識庫

  所有服務 → Coolify(共用 Docker 基礎設施)
```

> **核心口訣**:一次登入、一套規範、一份資料、一個部署平台、一個知識庫。

## 五大核心模組

| # | 模組 | 對應痛點 |
| --- | --- | --- |
| 1 | 統一登入入口(Microsoft AD 登入器) | 帳號各自為政 |
| 2 | 開發標準規範 | 系統長得不一樣 |
| 3 | 企業中央資料中心(Data Hub) | 資料重複維護 |
| 4 | Coolify 部署管理 | 雲端成本失控 |
| 5 | AI RAG 知識庫 | AI 做不起來 |

後續逐一展開每個模組。

## 模組 1 — 統一登入入口

**是什麼**
以 Microsoft AD 為單一身分來源,自建**登入閘道**代理所有系統驗證。

**為什麼**
- 員工離職,關一處全公司失效
- 權限集中管理,角色由 AD 管
- 減少資安面積,AD 帳密不散落

**關鍵設計**
- 協議:OIDC / SAML(視 AD 能力)
- Session 儲存:Redis,TTL 對齊 AD 政策
- 單一登出:中央撤銷後,各系統下次請求即 401

> 所有新系統**一律**整合此閘道,不得自建帳號系統。

**是什麼**

## 模組 3 — 企業中央資料中心(Data Hub)

**是什麼**
共用的**主檔資料**集中維護於一處,其他系統**只能讀、不能複製**。

**主檔範圍**

| 類別 | 範例 | 維護單位 |
| --- | --- | --- |
| 組織 | 公司、部門、成本中心 | HR / 財務 |
| 人員 | 員工、在職狀態、權限群組 | HR + AD 閘道 |
| 客戶 | 主檔、聯絡人、信用額度 | 業務 + 財務 |
| 代碼表 | 幣別、國家、付款條件 | 財務 |
| 會計 | 科目、稅別、發票類型 | 財務 |
| 商品 / 物料 | 品項、BOM、價格階層 | 採購 / 業務 |

---

## 模組 3 — 技術選擇與規範

**技術骨幹**
- 儲存:PostgreSQL + Adminer
- 對外:統一的主檔 REST 服務
- 變更推播:CDC / 事件訂閱
- 對接舊系統:ETL(涵蓋既有 SaaS 雙向同步)

**架構規範**
- 業務系統**只讀**主檔,禁止自建重複欄位
- 主檔語意變更必須**升版 + 過渡期**
- 所有寫入走 audit log,具可追溯性

---

## 模組 4 — Coolify 部署管理

**是什麼**
所有自建系統的**唯一部署平台**,Docker Compose 為部署單位。

**Coolify 負責**
- Build、網域、HTTPS、環境變數管理
- Log 集中、健康檢查、一鍵回滾
- 多服務共用一台機器(共用基礎設施)

**標準服務**(v1.2 起納入):Postgres + Adminer + Seq

> 完整配置規範見 `Docker-Compose-Spec-v1.2.md`。

---

## 模組 5 — AI RAG 知識庫

**是什麼**
Agentic AI 的知識基座,資料來自 Data Hub + 業務摘要 + 文件庫 + Seq log。

**技術設計**

| 項目 | 選擇 |
| --- | --- |
| Vector DB | pgvector(共用 Postgres)或 Qdrant |
| Embedding | OpenAI / Voyage / 本地 BGE |
| 推論 | Claude / GPT / 本地 |
| 索引觸發 | Data Hub CDC + 文件庫 webhook |
| 權限層 | 依 AD 群組 row-level filtering |

**四大鐵律**
1. 髒資料不索引 — 進庫前清洗與語意標註
2. PII 脫敏 — 電話 / 身分證 / 內部 IP 不入向量
3. 每筆回答附來源 — UI 必顯示 citation
4. 權限不得繞過 — AI 僅能檢索有權資料

## 根本問題 1 — 各系統重複建資料表

**現象**
員工、權限、客戶、幣別在不同系統各自維護,改一邊忘一邊。

**根因**
過去無中央主檔,每個團隊起專案就複製 schema。

**對策**
- Data Hub 為**唯一寫入來源**
- 規範明訂禁止重複定義主檔欄位
- `/scan-project` 擴充偵測違規

**成功指標**
- 新系統啟動,主檔類欄位 **0 複製**
- 員工離職,**1 小時內**全系統權限失效

---

## 根本問題 2 — 雲端成本偏高

**現象**
流量極低的工具獨佔一台 VM,整月 idle 仍計費。

**根因**
- 「一系統 = 一 VM」的預設架構
- 缺乏統一部署平台做資源共用

**對策骨架**(詳見後續成本章節)
- 統一上 Coolify,共用機器
- 資源邊界(CPU / memory limits)
- 下線機制:半年未用主動歸檔

---

## 雲端成本是「架構問題」

雲端成本**不是財務議題,是部署模式問題**。

多數浪費不來自「算得多」,而是**長尾 idle 資源累加**:
- 每台 VM 即使 0 流量也 24×7 計費
- 系統越多,常駐費用**線性增加**
- 跟實際使用率**無關**

> 省錢 ≠ 砍一台大機器
> 省錢 = **改變部署模式**

---

## 常駐費用拆解

每新增一個系統,疊加的**常駐費用**:

| 項目 | 0 流量是否付費 |
| --- | --- |
| VM 常駐費 | ✅ 付 |
| 區塊儲存 | ✅ 付 |
| 固定 IP / LB | ✅ 付 |
| 備份快照 | ✅ 付 |
| 監控 / agent 授權 | ✅ 付 |
| 流量費用(egress) | ❌ 不付 |

**結論**:真正浪費的是「什麼都沒算也要付」的部分。

## 部署模式對照

> **公司現況限制**:無 K8s 基礎建設、Serverless 僅限 GCP/AWS。

| 模式 | 成本(低) | 運維複雜度 | 我們能用嗎 |
| --- | --- | --- | --- |
| 一系統一 VM(現狀) | 🔴 高 | 🟡 中 | 最貴 |
| Coolify 共用(目標) | 🟢 低 | 🟢 低 | ✅ 主力 |
| Serverless (GCP/AWS) | 🟢 極低 | 🟢 低 | ⚠️ 限特定場景 |
| Replit / 託管 | 🟢 低 | 🟢 極低 | ✅ 個人工具 |
| ~~Kubernetes~~ | — | 🔴 高 | ❌ 無基礎建設 |

---

## Serverless 的正確用法

僅限 **GCP Cloud Run / AWS Lambda**,且只在以下情境:

| ✅ 適合 | ❌ 不適合 |
| --- | --- |
| Webhook 接收器 | 有 session 的 web app |
| 定時批次 / ETL | 長連線(WebSocket、SSE) |
| 偶發流量(< 幾百次 / 日) | 穩定流量(會吃冷啟動) |
| 對外靜態 + 少量 API | 內部系統(走 Coolify 即可) |

## 成本類別與治理手段

| 類別 | 範例 | 治理 |
| --- | --- | --- |
| **常駐費** | VM、DB、儲存 | 共用 + 排程啟停 |
| **流量費** | egress、API 呼叫 | CDN + 快取 |
| **儲存費** | 舊 log、備份 | TTL + 冷熱分層 |
| **建置費** | CI build、image pull | cache 層序、lock file |
| **外部服務** | OpenAI tokens、SMS | 預算上限 + 告警 |

---

## 優化手法(依影響力)

1. **共用基礎設施**(最大) — 多服務一台機、共用 Postgres(schema 隔離)、共用 Seq
2. **分層部署策略** — Tier 1 / 2 / 3 配不同資源
3. **資源邊界** — 每 service 設 CPU / memory limit
4. **排程啟停** — 非上班時段關 Tier 2/3
5. **儲存分層** — log 30 天、備份冷存、上傳 90 天後轉 IA
6. **CDN / 靜態資源** — 前端走 CDN,origin 只處理 API
7. **監控告警** — Billing 閾值、pay-per-token 預算硬上限

---

## 分層部署策略

| Tier | 範例 | 資源策略 |
| --- | --- | --- |
| **T1 — 生產關鍵** | 業務核心系統、財務、對客 | 獨立資源、完整監控、SLA 嚴格 |
| **T2 — 內部工具** | HR 簽核、報表 | 共用 Coolify、共用 DB(schema 隔離) |
| **T3 — 實驗 / 個人** | Vibe Coding、POC | 最小規格、排程啟停、自動下線 |

**典型結果**:原本 5 台 VM(2vCPU/4GB)合併成 1 台 4vCPU/16GB,帳單降 ~70%。

---

## Trade-off(要放棄什麼)

| 選擇低成本,換的是... | 對策 |
| --- | --- |
| 共用機器 = 單點故障放大 | 標準化備份 + 1 小時內重拉 |
| 共用 DB = noisy neighbor | 資源邊界、慢查詢監控、T1 獨立 |
| 排程啟停 = dev 體驗差 | 排程外可手動喚醒 |
| 壓太緊 = 高峰當機 | 保留 20-30% 緩衝,監控 p95 |
| 下線機制 = 可能誤殺 | 預告期 + owner 確認 |

**原則**:**T1 絕不為省錢妥協穩定**;T2/T3 積極節流。

---

## 成本治理原則

- **誰建的誰負責** — 上線時登記擁有者、預期流量、預期成本、生命週期
- **季度檢視** — 每季列 top N 高成本服務,檢視合理性
- **主動下線** — 半年無流量 → 預告一個月後下線
- **預算上限** — 各 Tier 月度總額,超過告警並凍結上線
- **決策留痕** — 跳脫標準要寫理由

## 情境 A — 新員工入職

```
1. HR 在 AD 建帳號、加入群組
2. AD 閘道同步新身分
3. Data Hub 寫入員工主檔
4. 員工首次登入業務系統:
   → AD 登入器驗證 → session
   → 業務系統讀 Data Hub 取角色
   → 顯示「業務部」可見範圍
5. Seq 收到登入事件 log
6. AI RAG 索引更新(角色可見範圍)
```

---

## 情境 B — 客戶主檔變更

```
1. 使用者於業務系統更新主檔資料
2. 業務系統呼叫 Data Hub API
3. Data Hub 寫入 + 發 CDC 事件
4. 訂閱者(ERP、收款、AI RAG)收到更新
5. AI RAG 重算 embedding,後續問答反映新額度
6. Seq 留完整 audit(誰 / 改什麼 / 時間 / requestId)
```

---

## 情境 C — 新系統上線

```
1. clone template → 依 spec 建骨架
2. sso-init 接上 AD 登入閘道
3. 接 Data Hub 讀主檔(自己 DB 只存業務)
4. 寫 docker-compose.yml(依 v1.2 規範)
5. Push → Coolify 自動 build / deploy
6. /scan-project 通過 → 納入正式營運
7. Log 進 Seq、資料進 AI RAG
```

## 建置順序與相依關係

```
[先建] Microsoft AD 登入器
   ↓
[同步] 開發標準規範(本倉庫持續維護)
   ↓
[接著] Data Hub(含 Postgres / 主檔)
[同步] Coolify 部署基礎建設
   ↓
[最後] AI RAG(需前面四項先穩)
```

**為什麼是這個順序**
- 登入沒有 → 無法做權限
- 規範沒有 → 各系統長得不一樣
- Data Hub 沒有 → AI 吃髒資料 = 失敗
- Coolify 沒有 → 成本壓不下
- AI RAG 放最後 → 前四穩了才值得做

## 一張投影片總結

- 公司缺一條**統一資料流**,每件事都在各自造輪子
- 解方是**五個統一**:登入、開發、資料、部署、AI
- 兩個根本痛點:**資料重複**、**成本失控**
- 部署走 **Coolify** 共用基礎設施,K8s 不在選項內

> 「**整合一次,長期收租**」— 這是整個藍圖的經濟學。
