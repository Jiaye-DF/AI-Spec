# 企業內部資料流整合 v1.0

> 定位:公司所有內部系統「**登入 → 開發 → 資料 → 部署 → AI**」的整合藍圖。
> 目標讀者:IT、各部門 IT 對口、後續要建置新系統的開發者。
> 互補文件:[`CLAUDE-v1.0.md`](../Development/Claude/CLAUDE-v1.0.md)、[`spec/00-overview-v1.0.md`](../Development/Project/spec/00-overview-v1.0.md)、[`Docker-Compose-Spec-v1.2.md`](../Coolify-Deploy/Docker-Compose-Spec-v1.2.md)、[`sso-init-*`](../Development/Project/sso-init/)

---

## 一、為什麼要這份藍圖

公司目前的系統彼此獨立成長,沒有一條貫通的資料流。造成的現象:

- **登入各做各的** — 每個系統自建帳號、自建權限
- **開發各做各的** — 同樣的需求,兩個團隊會做出兩套相似但不相容的東西
- **資料各存各的** — 客戶、員工、幣別等基礎資料在多處被複製維護
- **部署各做各的** — 每套系統一台 VM,流量極低也是整台機器開著
- **AI 想做做不了** — 資料散亂、格式不一,RAG 無法有效索引

這份文件把上述五件事連成**一條資料流**,讓新系統啟動時知道「每一段」該接到哪裡、用哪套規範、產物放哪。

## 二、整體資料流總覽

```
               ┌────────────────────────────────────┐
               │   Microsoft AD(單一身分來源)      │
               └────────────────┬───────────────────┘
                                │
                                ▼
               ┌────────────────────────────────────┐
               │   統一登入閘道 (AD 登入器)         │
               │   — token 簽發 / session / logout  │
               └────────────────┬───────────────────┘
                                │ 各系統憑 session 呼叫
                                ▼
 ┌─────────────────────────────────────────────────────────┐
 │    業務核心項目系統(遵循 AI-Spec 開發規範)             │
 │    ERP / CRM / BPM / 跨部門開發工具 ...                  │
 └────┬─────────┬─────────┬─────────┬──────────────────────┘
      │ 讀寫    │ 讀寫    │ 讀寫    │ 讀寫
      ▼         ▼         ▼         ▼
 ┌─────────────────────────────────────────────────────────┐
 │       企業中央資料中心(Data Hub / Master Data)        │
 │  員工 / 權限 / 客戶 / 幣別 / 組織 / 代碼表 / 會計科目    │
 └────────────────────┬───────────────────┬────────────────┘
                      │ 匯出 / Stream     │
                      ▼                   ▼
         ┌──────────────────────┐  ┌────────────────────┐
         │  Seq(結構化 log)    │  │  AI RAG 知識庫     │
         │  觀測 / 除錯         │  │  Agentic AI 使用   │
         └──────────────────────┘  └────────────────────┘

         所有服務皆部署於 ─── Coolify(共用 Docker 基礎設施)
```

> 核心口訣:**一次登入、一套規範、一份資料、一個部署平台、一個知識庫。**

## 三、五大核心模組

### 3.1 統一企業登入平台入口(Microsoft AD 登入器)

**是什麼**
以 Microsoft AD 為單一身分來源,由一支自建的**登入閘道**代理所有系統的驗證流程;各業務系統不自行連 AD,只認閘道發出的 session。

**為什麼要這樣做**
- **單一身分來源** — 員工離職停用一次,全公司系統立即失效
- **權限集中管理** — 角色 / 群組由 AD 管,新系統只需對應角色
- **減少資安面積** — AD 帳密不需重複散落於各業務系統

**模組內涵**

| 項目 | 說明 |
| --- | --- |
| 身分來源 | Microsoft AD(on-prem 或 Entra ID) |
| 協議 | OIDC / SAML(視 AD 能力選一),內部系統統一走閘道 |
| Session 儲存 | Redis,TTL 與 AD 政策對齊 |
| 單一登出 | 中央撤銷後,各系統下一次請求即 401 |
| 對開發者介面 | 依後端框架提供 `sso-init-*` 模板(見 [sso-init/](../Development/Project/sso-init/)) |

**交付物**
- AD 登入閘道服務(獨立部署於 Coolify)
- 前後端整合範本(Next.js / FastAPI / Spring)
- 整合文件(帳號申請 → `app_id` / `app_secret` 發放 → callback 對接)

**行為規範**
- 所有新系統**一律**整合此閘道,不得自建帳號系統
- 開發環境可用 mock,但必須能一鍵切回正式閘道
- 首頁 / 未授權行為遵循 [`spec/00-overview-v1.0.md`](../Development/Project/spec/00-overview-v1.0.md) 的五大原則

**是什麼**
所有公司自建系統在「架構、API、資料庫、部署、資安、Log」這些面向的共同底線,已整理於本倉庫。新專案不需重新想,套規範起骨架即可。

**規範地圖**

| 專案骨架與路由原則 | [`spec/00-overview-v1.0.md`](../Development/Project/spec/00-overview-v1.0.md) |

| SSO 整合 | [`sso-init-*`](../Development/Project/sso-init/) |

**為什麼**
- 新系統上線時間從「重新設計架構」縮短為「套規範 → 寫業務邏輯」
- 跨系統的 API 外殼、錯誤處理、Log 格式統一,Seq / AI 才能集中分析

**行為規範**

- `/api/v{n}` 版本前綴強制;路由與錯誤分類見 `spec/01-design-notes`

### 3.3 企業中央資料中心平台(Data Hub)

**是什麼**
把各系統共用的「**主檔資料**」集中維護於一處,其他系統一律**讀**這裡、不重複定義。

**哪些資料屬於主檔**

| 類別 | 範例 | 誰負責維護 |
| --- | --- | --- |
| 組織 | 公司、部門、職位、成本中心 | HR / 財務 |
| 人員 | 員工基本資料、在職狀態、權限群組 | HR + AD 閘道 |
| 客戶 | 客戶主檔、聯絡人、信用額度 | 業務 + 財務 |
| 代碼表 | 幣別、國家、單位、付款條件 | 財務 |
| 會計 | 科目、稅別、發票類型 | 財務 |
| 商品 / 物料 | 品項、BOM、價格階層 | 採購 / 業務 |

**為什麼**
- 避免「各部門各開一張員工表」的亂象
- AI RAG 吃的是乾淨主檔,不是髒業務資料
- 對帳 / 報表口徑一致,減少對齊工時

**技術選擇(建議)**

| 項目 | 選擇 | 說明 |
| --- | --- | --- |
| 儲存 | PostgreSQL + Adminer | Coolify 標準棧,已含於 Docker-Compose-Spec v1.2 |
| 對外 API | REST `/api/v1/master/*` | 其他系統讀取主檔的唯一通道 |
| 資料變更 | CDC / outbox + message bus | 讓下游系統訂閱變更事件(若需即時同步) |
| 對接既有 SaaS | ETL Job(排程 / event-driven) | 舊系統過渡期雙向同步(例:取代中的 SaaS 平台) |

**行為規範**
- 業務系統**禁止**在自己的 DB 重複存主檔欄位(需要時 snapshot,不可直接 writable)
- 主檔欄位語意變更必須**升版 + 保留過渡期**(對應 `/api/v{n}` 版本策略)
- 任何寫入必經 **audit log**(誰改、改前改後、時間)

### 3.4 Coolify 部署管理

**是什麼**
公司所有自建系統的**唯一部署平台**,以 Docker Compose 為部署單位,Coolify 負責 build、網域、HTTPS、環境變數、log 集中、健康檢查。

**為什麼**
- 取代「一套系統一台 VM」,多個服務可共用一台機器
- 低流量服務不必佔整機資源,直接 sleep 或最小化部署
- 環境變數、secret、網域設定都在後台,不用 SSH
- 回滾 / 重部署一鍵完成,**不需等 IT 進機器**

**配置規範**

詳見 [`Docker-Compose-Spec-v1.2.md`](../Coolify-Deploy/Docker-Compose-Spec-v1.2.md),重點:

- `docker-compose.yml`(副檔名 `.yml`,禁 `.yaml`)
- **不要** 定義 `networks` 區塊(Coolify 自動管理)
- **不要** 用 `${VAR:?error}` 語法
- `environment` 一律用 `KEY: value` map 語法,禁用 `- KEY=value` list
- Volume 必須**命名**,`latest` tag 禁用
- 標準服務:Postgres + Adminer + Seq(v1.2 起納入)

**命名與資源**

| 項目 | 規則 |
| --- | --- |
| Image tag | 綁 git SHA 或 semver,不用 `latest` |
| Service 命名 | 小寫短名,對應 `SERVICE_URL_{NAME}_{PORT}` |
| Volume 命名 | `${COMPOSE_PROJECT_NAME}-{用途}-data` |
| Secret | 一律走 Coolify 後台 env,禁寫死於 compose |

**行為規範**
- 新系統從 template 改起,不從零寫 compose
- 部署前跑 `/scan-project` 檢查 `R-DEP-*`
- `/health` 回 200 才視為上線成功;Seq 收到新版 log 才算真的通

### 3.5 AI RAG 知識庫建立

**是什麼**
公司內部的 Agentic AI 助理所依賴的知識基座,資料來自:
1. Data Hub 主檔(員工、客戶、代碼)
2. 各業務系統的交易摘要(匿名化 / 權限化後)
3. 文件庫(SOP、合約、技術文件)
4. Seq log 的營運脈絡(非 PII)

**為什麼**
- 讓 AI 回答的每一句話**可追溯來源**
- 把「問業務問題 → 翻系統 → 翻 Excel」變成「問 AI 一次就到」
- 把流失的 know-how(離職員工帶走的知識)鎖在公司內

**模組內涵**

| 項目 | 說明 |
| --- | --- |
| Vector DB | pgvector(與主檔共用 Postgres)或 Qdrant |
| Embedding 模型 | 視預算選:OpenAI / Voyage / 本地 BGE |
| 推論模型 | 依情境挑 Claude / GPT / 本地模型 |
| 索引觸發 | Data Hub CDC + 文件庫 webhook |
| 權限層 | 依 AD 群組過濾檢索結果(Row-level security) |

**行為規範**
- **髒資料不索引** — 進知識庫前經清洗與語意標註
- **PII 脫敏** — 電話 / 身分證 / 內部 IP 不入向量
- **每筆回答附來源** — UI 必顯示 citation,讓使用者能點回原檔
- **權限不得繞過** — AI 僅能檢索使用者有權看的資料

## 四、待解決的根本問題

### 問題 1 — 各系統重複建立資料表

**現象**
員工、權限、客戶、幣別等相同資料在不同系統各自維護,導致長期不一致、改一邊忘一邊。

**根因**
過去沒有中央主檔,每個團隊起專案就複製一份 schema。

**對策**
- 模組 3.3(Data Hub)成立為**唯一寫入來源**
- 規範 3.2 明訂「業務系統禁止重複定義主檔欄位」
- 透過 `/scan-project` 的自訂規則偵測違規(未來版)

**成功指標**
- 新系統啟動時,主檔類欄位 0 複製
- 員工離職後,所有系統存取權 1 小時內失效

---

### 問題 2 — 雲端機器成本偏高

**現象**
流量極低的工具也獨佔一台 VM,整月幾乎 idle 卻仍計費。

**根因**
- 過去以「一系統 = 一 VM」為預設架構
- 缺乏統一的部署平台做資源共用

**對策**(詳細拆解見第五章「雲端成本架構」)
- 所有系統統一上 **Coolify**(模組 3.4),多服務共用同一台機器
- 低流量服務降規 / 合併部署
- 設定資源上限(CPU / memory limits),避免單服務吃滿整機
- 建立「**下線機制**」 — 半年未使用的工具主動歸檔

**成功指標**
- 單台機器平均承載服務數 ≥ 5
- 月雲端帳單年對比下降(設定具體百分比依基線決定)

## 五、雲端成本架構

> 本章展開「問題 2」的成本設計細節。讀者:要做預算 / 決策部署模式的人。

### 5.1 為什麼要把成本當成「架構問題」

雲端成本大多不是被單一貴的服務吃掉,而是**長尾**的 idle 資源累加而成。
這意味著:**省錢不是砍一台大機器,而是改變部署模式**。因此成本屬於架構決策,不是財務議題。

---

### 5.2 現況成本拆解(痛點)

以目前「一系統一 VM」的模式,每新增一個系統會疊加以下成本:

| 項目 | 特性 | 即使 0 流量是否付費 |
| --- | --- | --- |
| VM 常駐費用 | 按小時計,大多 24×7 開著 | ✅ 付 |
| 區塊儲存(磁碟) | 按容量計 | ✅ 付 |
| 固定 IP / LB | 按個計 | ✅ 付 |
| 備份快照 | 按容量計 | ✅ 付 |
| 監控 / agent 授權 | 多數按端點計 | ✅ 付 |
| 流量費用(egress) | 按 GB 計 | ❌ 不付 |

> 結論:**系統越多 → 常駐費用線性增加 → 與實際使用率無關**。真正浪費的不是「算得多」,是「什麼都沒算也要付」。

---

### 5.3 幾種部署模式的成本/運維對照

> ⚠️ **本公司 DevOps 現況限制**
> - **沒有 Kubernetes 基礎建設**(不具備 K8s cluster、也無專職維運人力)
> - **Serverless 僅限於 GCP / AWS 環境**(其他雲或地端無此能力)

| 模式 | 成本(低流量) | 成本(高流量) | 運維複雜度 | 我們能用嗎 |
| --- | --- | --- | --- | --- |
| **一系統一 VM**(現狀) | 🔴 高 | 🟡 中 | 🟡 中 | ✅ 可,但最貴 |
| **共用 VM + Coolify**(目標) | 🟢 低 | 🟡 中 | 🟢 低 | ✅ 本公司主力路線 |
| **Serverless(GCP Cloud Run / AWS Lambda)** | 🟢 極低 | 🔴 可能失控 | 🟢 低 | ⚠️ 僅限 GCP / AWS 專案、且流量型態合適時才用 |
| **Replit / 託管服務** | 🟢 低 | 🔴 擴展受限 | 🟢 極低 | ✅ 個人工具、小組實驗 |

**為什麼落在 Coolify**
- **K8s 不在選項內** — 沒 cluster、沒人維運,強上只會變另一個技術債
- **Serverless 只能在特定場景用** — 專案必須部署在 GCP/AWS,且流量偶發、無長連線 / session 的服務才適合(例:批次任務、webhook receiver、靜態前端)
- **一 VM 一系統** — 已證實成本最高,繼續走只會擴大問題 2
- **Coolify** — 對公司目前規模(多系統、低流量為主)是**成本與運維複雜度都最平衡**的選擇

**Serverless 在我們情境下的建議用法**

| 適合用 Serverless(GCP/AWS 限定) | 不適合用 |
| --- | --- |
| Webhook 接收器、事件處理 | 有 session / DB 連線池的 web app |
| 定時批次(報表、ETL 排程) | 需要常駐連線(WebSocket、SSE) |
| 對外公開的靜態前端 + 少量 API | 內部系統(走 AD 閘道、Coolify 生態即可) |
| 流量極偶發(一天 < 幾百次) | 穩定流量,會吃到冷啟動懲罰 |

---

### 5.4 成本類別(分類才治得住)

| 類別 | 範例 | 治理手段 |
| --- | --- | --- |
| **常駐費用** | VM、DB、儲存、固定 IP | **共用 + 排程啟停** |
| **流量費用** | 對外 egress、API 呼叫、影像下載 | **CDN + 快取** |
| **儲存費用** | 舊 log、備份、檔案 | **TTL + 冷熱分層** |
| **建置費用** | CI/CD build minutes、image pull | **cache 層順序 + lock file** |
| **外部服務** | OpenAI tokens、email、SMS | **預算上限 + 告警** |

---

### 5.5 優化手法(依影響力排序)

#### 1. 共用基礎設施(影響力最大)

- 所有內部系統上 Coolify,一台機器跑多個 container
- 共用 Postgres,以 **schema 隔離**(非 database per service)
- 共用 Seq(v1.2 起納入標準服務)
- 共用 Adminer / 備份腳本

> 一個典型結果:原本 5 台 VM(每台 2vCPU/4GB)合併成 1 台 4vCPU/16GB,帳單下降 ~70%。

#### 2. 分層部署策略(依重要性配資源)

不是所有系統都需要 24×7 生產級資源。

| 分層 | 範例 | 資源策略 |
| --- | --- | --- |
| **Tier 1 — 生產關鍵** | 業務核心系統(ERP/CRM/BPM 等)、財務、對客系統 | 獨立或半獨立、完整監控、SLA 嚴格 |
| **Tier 2 — 內部工具** | HR 簽核、報表、內部儀表板 | 共用 Coolify、共用 DB(schema 隔離) |
| **Tier 3 — 實驗 / 個人** | Vibe Coding 產物、POC | 最小規格、可排程啟停、可自動下線 |

#### 3. 資源邊界(避免 noisy neighbor)

共用機器最大的風險是**一個服務吃滿整機**。Docker Compose 一律設定:

```yaml
services:
  backend:
    deploy:
      resources:
        limits:
          cpus: "1.0"
          memory: 1G
        reservations:
          memory: 256M
```

- `limits`:硬上限,超過被 kill
- `reservations`:保留最低量,確保啟動可用

#### 4. 排程啟停(適用 Tier 2/3)

- 開發 / 測試環境 **非上班時段 / 週末自動關**
- 報表服務只在月初 / 週五需要 → 其他時間 scale to 0
- Coolify 支援手動 start/stop,自動化可透過 GitHub Actions + Coolify API

> 以 40 小時 / 168 小時計,非 24×7 服務可省 **~76%** 常駐費用。

#### 5. 儲存與 log 分層

| 資料 | 預設保留 | 策略 |
| --- | --- | --- |
| Seq 應用 log | 30 天 | 熱存查詢,超過歸檔至冷儲存 |
| DB 備份 | 7 日增量 + 4 週週期 | 週期備份移 S3 Glacier 等 |
| 使用者上傳 | 永久 | 熱存 90 天後轉 IA |
| CI build artifact | 14 天 | 超過自動刪 |

#### 6. CDN / 靜態資源

- 前端 build 產物(hash 檔名)一律走 CDN,`Cache-Control: max-age=31536000, immutable`
- HTML index `Cache-Control: no-cache`(才能拉到新版 hash)
- 對外流量由 CDN 吸收,origin 只處理 API

> 參考 `spec/01-design-notes-v1.0.md` 的「靜態資源快取」章節。

#### 7. 監控告警(不讓成本暴衝)

- 雲端 Billing 設**每日 / 每月閾值告警**
- OpenAI 等 pay-per-token 服務設**硬性預算上限**,超過直接拒呼叫
- 流量異常成長(爬蟲、誤迴圈)要在 1 小時內被發現

---

### 5.6 成本治理原則

- **誰建的誰負責** — 服務上線時登記:擁有者、預期流量、預期月度成本、生命週期
- **季度檢視** — 每季列出 top N 高成本服務,檢視是否合理
- **主動下線** — 半年無流量或擁有者離職且未交接 → 預告一個月後下線
- **預算上限** — 每個 Tier 訂月度總額,超過告警並凍結新服務上線
- **決策留痕** — 跳脫標準(例如 Tier 2 要獨立 DB)要寫理由,方便後續追溯

---

### 5.7 Trade-off(老實說要放棄什麼)

| 選擇低成本路線,換的是... | 對策 |
| --- | --- |
| 共用機器 = 單點故障放大 | 標準化備份 + 快速重建(1 小時內重拉全站) |
| 共用 DB = noisy neighbor 風險 | 資源邊界、慢查詢監控、必要時 Tier 1 獨立 |
| 排程啟停 = dev 體驗略差 | 排程外可手動喚醒,週一早上自動開 |
| 資源壓太緊 = 高峰當機 | 保留 20-30% 緩衝,監控 p95 而非平均值 |
| 下線機制 = 可能誤殺小工具 | 預告期 + owner 確認機制 |

**核心原則**:Tier 1 絕不為了省錢妥協穩定;Tier 2/3 積極節流。

---

### 5.8 成本目標與檢核方式

**目標(需先量基線才能定具體數字)**

| 指標 | 目標方向 |
| --- | --- |
| 單台機器承載服務數 | ≥ 5(Tier 2/3) |
| 月雲端帳單年對比 | 下降(具體 % 依基線) |
| 每新系統上線的邊際成本 | 接近 0(Tier 3 應如此) |
| Idle 資源占比 | < 30%(24h 平均) |

**檢核**

- 每月匯出雲端 Billing 明細,與 Coolify 服務清單對帳
- `/scan-project` 未來版本將增加「Docker 無資源限制」警示
- 預設**每季**與 IT 主管討論 Tier 結構是否需要調整

## 六、資料流情境實例

### 情境 A — 新員工入職

```
1. HR 在 AD 建立帳號,加入「業務部」群組
2. AD 閘道感知新身分(或下次登入時同步)
3. Data Hub 透過 HR 系統寫入員工主檔
4. 員工首次登入業務系統:
   - 導向 AD 登入器 → 驗證通過 → session
   - 業務系統讀 Data Hub 取得員工資料 / 角色
   - 顯示「業務部」可見的資料範圍
5. Seq 收到登入事件 log
6. AI RAG 索引更新(角色可見範圍擴張)
```

### 情境 B — 主檔變更

```
1. 使用者於業務系統更新主檔資料
2. 業務系統呼叫 Data Hub /api/v1/master/:resource
3. Data Hub 寫入 + 發出 CDC 事件
4. 訂閱此事件的系統(其他業務系統、AI RAG)收到更新
5. AI RAG 重算 embedding,後續問答反映最新狀態
6. Seq 紀錄完整 audit(誰改、改什麼、時間、requestId)
```

### 情境 C — 新系統上線

```
1. 工程師 clone template → 依 spec/00-overview 建骨架
2. 執行 sso-init-* 接上 AD 登入閘道
3. 接 Data Hub 讀主檔,自己的 DB 只存業務資料
4. 寫 docker-compose.yml(依 Docker-Compose-Spec v1.2)
5. Push → Coolify 自動 build / deploy
6. /scan-project 通過 → 納入正式營運
7. Log 進 Seq;主檔 / 文件進 AI RAG
```

## 七、相依關係與建置順序

```
(先建) Microsoft AD 登入器  ─────────┐
                                       │
(同步) 開發標準規範(本倉庫持續維護)── 各模組都會引用
                                       │
(接著) Data Hub(含 Postgres / 主檔) ─┤  新系統需先有登入
                                       │  + 規範才建得起
(同步) Coolify 部署基礎建設 ──────────┤
                                       │
(最後) AI RAG(需前面四項先穩) ───────┘
```

**為什麼是這個順序**
- 登入沒有 → 任何系統都做不了權限控制
- 規範沒有 → 每個系統長得不一樣,後面難整合
- Data Hub 沒有 → AI 吃髒資料 = 失敗
- Coolify 沒有 → 成本壓不下來,系統越多越爛
- AI RAG 放最後 → 前四項穩了才值得做

## 八、階段性交付建議

| 階段 | 核心交付 | 預期產出 |
| --- | --- | --- |
| Phase 0 | 本藍圖 + 規範共識 | 各部門 IT 對口同步 |
| Phase 1 | AD 登入閘道 + `sso-init-*` 模板 | 新系統可套整合 |
| Phase 2 | Data Hub MVP(員工 + 客戶 + 代碼表) | 業務核心系統可只讀主檔 |
| Phase 3 | Coolify 標準化 + Seq 上線 | 所有新系統部署一致 |
| Phase 4 | `/scan-project` 擴充 Data Hub 規則 | 自動偵測重複主檔 |
| Phase 5 | AI RAG MVP + 文件庫整合 | AI 助理接上 Data Hub |
| Phase 6 | 舊系統收斂(重複資料遷移、成本優化) | 老系統納入 Coolify |

> 實際時程依資源評估,本文件只標示**先後邏輯**,不綁定日期。
