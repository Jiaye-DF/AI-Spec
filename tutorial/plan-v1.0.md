# DF 內部 Vibe Coding 教學課程 · 計畫書 v1.0

> 12 週 × 1 小時 · **每週獨立 workshop** · 不設結業 · 對象：完全新手非工程師

---

## 一、課程目標

讓**完全新手的非工程師**，在 12 週內初步建立 AI 協作的基本素養：

1. **AI 協作思維** — 知道怎麼跟 AI 一起做事，而不是把 AI 當「更快的工人」
2. **治理基本觀念** — 理解為什麼 AI 寫的東西不能直接用，審查機制是在保護誰
3. **能跟工程師對話** — 聽得懂工程師在說什麼（伺服器、部署、PR、git），知道自己在整個流程裡的位置

**課程使命**：讓非工程師同仁建立 AI 協作時代的基本素養 — 不一定要變成工程師，但能跟工程師對話、能跟 AI 合作、能避開常見的坑。

## 二、學員定位與關鍵假設

| 項目 | 描述 |
|------|------|
| 學員程度 | **完全新手非工程師**，正要開始學習這些新知識 |
| 已有背景 | 會用電腦、會打字、可能聽過 ChatGPT 但不清楚 AI 協作是什麼 |
| 出席模式 | **每週可能是不同人**，允許自由參加 |
| 結業概念 | **不設結業** — 每週的價值就是該週本身 |
| 講師 | 課程規劃者本人 |
| 教具策略 | 用通用素材（日常 App、公開 GitHub 專案），不依賴特定平台 |

## 三、設計哲學

### 三大原則

1. **每週自成一體** — 開場 5 分鐘重建 context，不依賴前次出席。任一週單獨參與皆可帶走完整收穫。

2. **矯正觀念，不教操作** — 學員已具 AI 使用經驗，缺口在判斷力而非熟練度。以學員匿名化的劣質 prompt 與 code 為主要教材，跳過基礎操作示範。

3. **痛感取自內部** — 內部已累積真實事故，共鳴強於業界新聞。每週開場故事優先採匿名化內部案例，業界事件為輔。學員定位為「事故當事人」，課程目標為協助識別自身位置。

## 五、各週課程細節

### W1 · 什麼是 AI 協作？為什麼學這個

| 項目 | 內容 |
|------|------|
| 開場故事 | 同事用了 AI 一週，完成原本要做兩週的事 |
| 主題教學 | • 「AI 協作」是什麼：不是「叫 AI 寫程式」，是從一開始就讓 AI 參與想、做、改<br>• 跟「偶爾用 AI」的差別<br>• 為什麼現在每個人都該學一點 |
| 能與不能 | AI 能做什麼（草稿、查資料、補完），不能做什麼（下決定、為錯誤負責）|
| 範例參考 | 一個真實的 AI 協作流程是什麼樣子 |
| 帶走的東西 | 看得懂 AI 協作跟「偶爾用 AI」的差別，知道接下來會學到什麼 |

### W2 · AI 思維：合作而非命令

| 項目 | 內容 |
|------|------|
| 開場故事 | 學員「叫 AI 做」結果做歪的真實案例 |
| 主題教學 | • 命令式心態 vs 合作式心態<br>• 為什麼 AI 不是「更快的工人」，是「需要對話的同事」<br>• 「我想清楚 → 跟 AI 講 → AI 回應 → 我再想」的循環 |
| 對照示範 | 同一需求，命令式 prompt vs 合作式對話的 AI 輸出差異 |
| 帶走的東西 | 知道為什麼自己跟 AI 講話會失敗，有具體的改善方向 |

### W3 · 為什麼 AI 寫的東西要被審查

| 項目 | 內容 |
|------|------|
| 開場故事 | AI 寫的 code 上線炸掉、追責找不到人的故事 |
| 主題教學 | • 治理 3 大支柱：**可追溯（誰做了）/ 可審查（對不對）/ 可回滾（出事能回頭）**<br>• 沒有治理的世界長什麼樣 — 漏洞遍地、出事沒人擔 |
| 淺白類比 | 銀行為什麼要對帳、合約為什麼要簽名、開源專案為什麼都要走 PR |
| 穿插議題 | 🔒 **資安**：沒治理 = 安全漏洞的最佳溫床 |
| 帶走的東西 | 能說出治理 3 大支柱，知道為什麼這對非工程師也重要 |

### W4 · Prompt 結構入門：目的 / 背景 / 限制 / 範例

| 項目 | 內容 |
|------|------|
| 開場故事 | 一個學員的「我講得很清楚啊！」結果 AI 完全做歪的例子 |
| 主題教學 | • 4 欄位模板：**目的 / 背景 / 限制 / 範例**<br>• 每個欄位的用途與常見錯誤<br>• 對照示範：模糊 vs 結構化的 AI 輸出差異 |
| 現場練習 | 學員把一個自己用過的爛 prompt 套模板重寫 |
| 帶走的東西 | 拿到一張「Prompt 模板速查卡」，下週起會用 |

### W5 · 從 AI 視角理解 prompt：為何這樣下有效

| 項目 | 內容 |
|------|------|
| 開場故事 | AI 「自信地撒謊」的真實例子（虛構函式、編造 API） |
| 主題教學 | • LLM 是什麼：不是「知道答案的字典」，是「**會接話的補完機器**」<br>• 為什麼 AI 會幻覺、會自信、會誤解你<br>• 知道機制後，prompt 的「目的 / 背景 / 限制 / 範例」為什麼有效 |
| 穿插議題 | 🤖 **AI 風險**：幻覺類型清單、怎麼抓出來、什麼情境下絕對不能信 AI |
| 帶走的東西 | 從機制理解 prompt，以後自己能判斷某種寫法為什麼有效或無效 |

### W6 · 系統架構入門：一個東西有哪些層

| 項目 | 內容 |
|------|------|
| 開場故事 | AI 寫了一個「能跑」但沒分層的程式，後續維護地獄 |
| 主題教學 | • 一個系統的常見層次：**呈現層（看得到）/ 業務邏輯（規則）/ 資料（存放）/ 基礎設施（跑在哪）**<br>• 為什麼分層 — 改一處不會炸別處 |
| 日常例子 | 用「外送 App」每層用生活語言講明（畫面 / 計價 / 訂單記錄 / 機房）|
| 現場練習 | 拆解一個學員熟悉的 App（銀行、購物、訂房）|
| 帶走的東西 | 看到任何系統能說出它的「層次」，跟工程師對話有共同語言 |

### W7 · 思維邏輯：需求 → 設計 → 實作

| 項目 | 內容 |
|------|------|
| 開場故事 | 學員「整包需求丟給 AI」結果做歪的案例 |
| 主題教學 | • 工程思維的 3 步：**理解需求 → 拆解設計 → 逐步實作**<br>• 為什麼跳過「拆解設計」一定會做歪<br>• 用 AI 協作的時候，「拆解設計」這一步該你做、不是丟給 AI |
| 現場練習 | 給一個真實需求（例：整理客戶資料寄信），全班一起拆步驟 |
| 穿插議題 | 🤖 **AI 風險**：把「設計決策」交給 AI 的危險 |
| 帶走的東西 | 拿到任何需求，知道怎麼拆成步驟再給 AI |

### W8 · 責任邊界：誰負責什麼、AI 負責什麼

| 項目 | 內容 |
|------|------|
| 開場故事 | 「AI 寫的啊，不是我寫的」這種推卸後出事的故事 |
| 主題教學 | • 即使是 AI 寫的 code，**送出去的是你的名字**<br>• 哪些決定 AI 可以做、哪些絕對不能（架構、安全、隱私、合規）|
| 穿插議題 | 🔒 **資安**：AI 寫的 code 出事，責任歸屬怎麼算（法律 / 職場 / 合規面）|
| 帶走的東西 | 一張「**不能交給 AI 全權的決定**」清單 |

### W9 · 本地端：你的電腦不等於世界

| 項目 | 內容 |
|------|------|
| 開場故事 | 「我電腦能跑啊」結果換台電腦就壞掉的經典故事 |
| 主題教學 | • 什麼是本地端 — 你的電腦、你的環境、你的執行<br>• 為什麼「我電腦能跑」不等於「會動」 — 環境差異、套件版本、作業系統、相對路徑<br>• 怎麼讓你的東西「在別處也能跑」的基本概念（環境變數、相依套件記錄）|
| 帶走的東西 | 知道本地能跑 ≠ 真的能用，以後寫東西會多想一步 |

### W10 · 伺服器端是什麼？部署觀念入門

| 項目 | 內容 |
|------|------|
| 開場故事 | 部署上去後「為什麼線上的跟本地的不一樣？」的故事 |
| 主題教學 | • 什麼是伺服器：24 小時運作、別人會連你、出事影響廣<br>• 部署的基本元素：**環境變數 / health check / build / rollback**（用淺白話講）<br>• 為什麼 secret 不能寫死 — 一旦 push 到 GitHub 全世界都看得到 |
| 部署工具 | 業界常見的部署平台（Vercel、Heroku、Coolify 等）在做什麼 — 概念帶過，不需要學員實際操作 |
| 穿插議題 | 🔒 **資安**：secret 外洩事件、API key 寫死的後果 |
| 帶走的東西 | 聽得懂工程師講「我來部署一下」是在幹嘛，知道 secret 為什麼要小心 |

### W11 · Git / GitHub 基本概念

| 項目 | 內容 |
|------|------|
| 開場故事 | 沒版控的人「改壞了回不去」毀掉一個下午的故事 |
| 主題教學 | • Git 是什麼：**有歷史的存檔系統**<br>• 三個核心概念：**commit（存檔點）/ branch（平行宇宙）/ merge（合併）**<br>• GitHub Desktop 操作示範，不碰命令列<br>• 不要懂全部，只要懂「我可以隨時回到上個版本」 |
| 現場練習 | 學員跟著做一次：改檔 → commit → 看歷史 → 復原 |
| 帶走的東西 | 會用 GitHub Desktop 做存檔點與回頭 |

### W12 · PR 流程：讓你的 code 被審、被合、被部署

| 項目 | 內容 |
|------|------|
| 開場故事 | 沒走 PR 直接推 main，出事沒人知道的故事 |
| 主題教學 | • PR 是什麼：**「請求合併」的提案 + 審查機制**<br>• 為什麼要 PR 不直接推 — 治理、追溯、品質<br>• 全流程：開 PR → 同事/AI review → 修 → 合併 → 部署 |
| 真實例子 | 看一個公開 open source 專案的 PR（例如 React 或 VS Code） |
| 穿插議題 | 🔒 **資安**：不走 PR 的後果 — 後門 code、secret 漏出去 |
| 帶走的東西 | 看懂任何 PR 的構造，理解審查為什麼是保護而非麻煩 |

## 六、每週 60 分鐘的時間結構

每週都用同一個結構，讓學員無論哪週來都熟悉節奏：

| 時段 | 時間 | 內容 |
|------|------|------|
| 開場：context + 失敗故事 | 10 min | 5 分鐘重新建立 context（不假設聽過前面）+ 5 分鐘失敗故事 |
| 主題教學 + 例子 | 30 min | 講師示範「壞做法 → 好做法」對照，用日常例子說明技術概念 |
| 現場練習 / 討論 | 15 min | 學員當場練，卡住的當場問 |
| 收尾：本週帶走的東西 | 5 min | 把本週的 takeaway 講第二次，發速查卡 |

**關鍵原則**：
- 不要假設學員聽過上週
- 每週的「帶走的東西」要明確、可立刻用
- 技術術語要翻譯成日常語言

## 七、資安 / AI 風險的穿插設計

不獨立成週，而是綁定到相關週次自然融入。**每個穿插約 5 分鐘**。

### 議題對照表

| 週次 | 穿插議題 | 5 分鐘要傳達的事 |
|------|---------|----------------|
| W3 | 🔒 沒治理 = 安全漏洞溫床 | 用真實事故說明治理不只是流程美感，是實質防護 |
| W5 | 🤖 LLM 幻覺類型 | 列出常見幻覺（虛構函式、編造 API、計算錯誤），教怎麼快速辨識 |
| W7 | 🤖 把設計決策交給 AI 的危險 | AI 不知道你的業務脈絡，架構決策不能丟給它 |
| W8 | 🔒 AI code 出事的責任歸屬 | 即使是 AI 寫的，送出去的是你的名字 — 法律、職場、合規責任都還在你身上 |
| W10 | 🔒 secret 外洩事件 | API key 寫死、push 到 GitHub 的真實後果（業界已有多起公開事件）|
| W12 | 🔒 不走 PR 的後果 | 後門 code、無追溯、合規問題 |

其他週次（W1、W2、W4、W6、W9、W11）沒固定穿插，但講師可以隨機帶到。

## 八、教具來源：不靠特定平台，用通用素材

對完全新手的非工程師，內部系統當教具反而會造成距離感。改用人人可理解的素材：

| 素材 | 用途 | 對應週次 |
|------|------|---------|
| 日常 App 拆解 | 講系統架構 | W6 |
| 公開 GitHub PR（React、VS Code 等）| 講 PR 流程是什麼 | W12 |
| 業界事故新聞 | 講治理、責任、secret 外洩 | W3、W8、W10 |
| 學員自己的爛 prompt / code | 匿名化後當教材 | W2、W4、W5 |
| 講師以前的爛 code | 開場破冰，降低防備 | W1、W2 |

**重要原則**：不引入特定公司平台 demo，因為：
- 非工程師看不懂自家內部系統長什麼樣
- 通用素材跨組織通用，課程價值更普遍
- 學員注意力應放在「概念」而不是「這個平台怎麼操作」

## 九、失敗故事庫（開課前要準備）

每週開場 5 分鐘的故事，是課程價值的關鍵。

### 故事類型 × 對應週次

| 類型 | 對應週次 | 範例方向 |
|------|---------|---------|
| 治理缺位導致的事故 | W3、W8、W12 | Production 掛掉、追責失敗、合規問題 |
| Secret / 密碼外洩 | W10 | API key 寫死、commit 進 repo、外人連到資料庫 |
| AI 寫出的問題 code | W5、W7、W8 | 幻覺函式、安全漏洞、把設計決策丟給 AI 的後果 |
| Prompt 模糊導致做歪 | W2、W4 | 「我以為講清楚了」結果 AI 做完全不一樣的東西 |
| 本地能跑線上不能 | W9、W10 | 換環境就壞、相對路徑、套件版本 |
| 沒版控的災難 | W11 | 改壞了回不去、覆蓋了同事的 code |

**收集任務**：講師整理 12 個真實故事（可匿名化），每個包含：**情境 / 損失 / 如果有治理會怎麼樣**。可以從業界公開事件、新聞、講師個人經驗、同事閒聊蒐集。

## 十、教學風格

### 核心原則：痛感先行，概念後到

直接講觀念**會被當官腔**。學員聽完故事，後面的規則就會自己接受 — 不是「老師說要這樣」，是「我不要變成故事裡那個人」。

### 對完全新手的設計要點

- **用日常例子** — 外送、銀行、Email 這類非工程師熟悉的事
- **翻譯技術術語** — 每個 jargon 都要有一個生活語言版本
- **不要假設前提** — 「commit 是什麼」、「PR 是什麼」、「伺服器是什麼」都從零講起
- **避免炫技** — 講師不要 demo 自己多會用 git command，反而會讓學員退縮

## 十一、講師備課策略

### 工作量預估

備課工作量約上課時間 3–5 倍 → 12 小時上課 = **36–60 小時備課**。

### 階段式準備，不一次寫完 12 週

因為每週獨立，可分階段準備、試教、調整：

1. **先備好 W1–W3** — 認識 AI 協作與治理觀念是課程入口
2. **試教 1–2 場** — 找非工程師同事當白老鼠，收回饋
3. **再寫 W4–W8** — Prompt 與系統思維主題
4. **最後 W9–W12** — 部署與工具

### 開課前必備教材清單

- [ ] **12 個真實失敗故事**（配對到週次）
- [ ] **5–10 個爛 prompt / 爛 code 樣本**（W2、W4、W5、W6 教材）
- [ ] **幾個公開 GitHub PR 範例**（W12 用，例如 React、VS Code）
- [ ] **GitHub Desktop 教學環境**（W11 用）
- [ ] **每週速查卡**（A4 一頁，本週重點，學員帶走）
- [ ] **講師自己以前的爛 code**（開場破冰用）

## 十二、風險與緩解

| 風險 | 影響 | 緩解 |
|------|------|------|
| 學員把治理當官腔 | 學完跟沒學一樣 | 全程「痛感先行」；每週故事不省 |
| 不同週學員程度差太多 | 講師難以拿捏深度 | 每週開場 5 分鐘重新校準 context；速查卡作為共同基準 |
| 學員不連續來，觀念片段 | 學員自己拼湊不出全景 | 接受這就是設計；若有需要，做一份「12 週全景圖」一頁海報 |
| 講師備課工作量爆 | 課程延期 | 階段式準備，先試教 W1 再寫後續 |
| 講師不小心用太多技術 jargon | 學員聽不懂、放棄 | 試教時請非工程師同事打斷講師，出現 jargon 就要當場翻譯 |
| 資安議題講不深 | 學員以為知道但不夠 | 課程定位是「意識覺醒」，不是專業培訓；資安專業另開進階班 |
