# 公司專案 開發 + 上線 問題

> 這份文件用來整理我們目前遇到的問題、背後原因,以及初步的解法方向。專業名詞會附上白話解釋,方便還不熟背景的同仁快速跟上。

---

## 一、平台怎麼分工?

| 要做什麼 | 用哪個平台 | 為什麼 |
| --- | --- | --- |
| 個人用的小工具、自動化腳本、效率網站 | **Replit**(自己開發、自己部署、自己使用) | 快、輕、不需要別人協助 |
| 跨部門用的系統、或要讓外部能連進來的服務 | **Coolify**(由 IT 統一管,走完整流程) | 正式、安全、好追蹤 |

🔑 **核心原則**:個人的小實驗歸個人,公司的正式服務歸正式流程,不要混在一起。這樣出問題才知道找誰、誰負責。

---

## 二、公司帳號登入(SSO)

> **SSO** = Single Sign-On,「單一登入」。像一張員工卡,刷一次就能進所有系統,不用每套系統都記一組帳密。

- 所有新專案**一律**接公司 SSO,不能自己做一套登入系統。
- 整合方式照這份官方文件:https://github.com/Dafon-IT/DF-SSO/blob/main/INTEGRATION.md
- 程式對外溝通一律用 **RESTful API**(一種標準的網路請求格式,像大家都講同一種語言)。

## 三、目前遇到的三個核心問題

### 問題 1:程式「搬家」時,重要設定常常掉在半路

> **環境變數(Environment Variable)**:程式執行時需要的設定值,例如資料庫網址、API 金鑰。這些不寫死在程式碼裡,而是從外面塞進去,才能在不同環境用不同值。

**發生什麼事**
- 在自己電腦上跑得好好的程式,一推到 Replit 或 Coolify 就壞掉。
- 原因:金鑰、連線字串這些設定沒有跟著搬過去,程式一啟動找不到,直接當機。

**解法方向**
- 專案裡放一份 `.env.example`,列出「需要哪些設定」(只寫名稱、不寫真正的值),並放進 git 讓大家都看得到。
- 真正帶密碼的 `.env` 檔案**絕對不進 git**(加進 `.gitignore`),避免密碼外洩。
- 程式一啟動就先檢查必要設定有沒有到齊,少一個就立刻報錯停掉(**fail fast**,寧可早死不要偷偷壞掉)。
- Replit 用它的 Secrets 面板設定、Coolify 用它的 Environment Variables 面板設定,README 裡寫清楚對照。

---

### 問題 2:開發站和正式站搞混,把 localhost 網址登記給外部服務

> **localhost** = 「本機」,指工程師自己這台電腦。只有自己能連,別人都連不到。
> **開發站**:工程師自己電腦上的環境,網址通常是 `http://localhost:xxxx`。
> **正式站**:部署在公司伺服器上、所有人都能連的環境,有正式對外網址。

**發生什麼事**
- 工程師開發時為了方便,把對外網址寫成 `http://localhost:3000` 之類的本機網址。
- 這種網址常常需要登記到外部服務(例如 SSO 回呼、第三方 webhook、金流通知)。
- 結果整套服務只有工程師自己那台電腦連得到,部署到正式站後其他人完全打不開。
- 更糟的是,外部服務以為設定正常,但回呼全部打到不存在的 localhost,壞了也沒人發現。

**解法方向**
- 所有對外網址**一律**用環境變數帶進程式,程式碼裡**絕對不寫死**(禁止 hardcode `localhost` / `127.0.0.1`)。
- 專案裡建立一個統一的 `config` 模組,所有設定都從這裡進出,其他地方不要直接讀環境變數。
- 註冊外部服務的 callback / webhook 時,**一律使用 Coolify 分配的正式對外網址**,不可以用開發者個人電腦的網址。
- 程式啟動時,印出目前的對外網址和連線目標,讓維運人員打開流量前能檢查一次。
- 部署完後,跑一個簡單的自動測試(**smoke test**,冒煙測試),確認外部服務能正確回呼正式網址才放行。

---

### 問題 3:Replit 和 Coolify 的「家規」不一樣,程式搬家常水土不服

**發生什麼事**
- Replit 的設定集中在 `.replit` 這個檔案裡,很多東西是平台幫你決定的。
- Coolify 則要求你自己提供環境變數,還要給一個**對外可連線的網址**(Public URL),否則服務接不起來。
- 兩邊的規矩不同,同一份程式常常這邊能跑、那邊不能跑。

**解法方向**
- 專案遵守 **12-Factor App** 原則(一套寫雲端服務的共通準則):所有會變的設定一律走環境變數,不要塞進 `.replit`、也不要寫死在程式裡。
- 用 **Docker** 把專案打包成一個「統一規格的盒子」。不管搬到哪個平台,盒子裡的環境都一樣,消滅「這邊能跑、那邊不能跑」的問題。
- Coolify 走 **Git 自動部署**:設定好之後,只要 push 到指定的分支就自動上線,不要再用手動搬檔案的方式。
- 對外網址**一律以 Coolify 分配的為準**。SSO 回呼、第三方 webhook,都不可以寫 Replit 的個人網址(因為隨時會消失)。

## 四、Replit 小工具 → Coolify 正式服務 升級清單

當某個原本自己用的 Replit 小工具,要變成全公司用的正式服務時,依序做完這幾步:

1. 把原始碼搬到公司的 Git。
2. 補上 `.env.example`(環境變數清單)和 `Dockerfile`(打包設定)。
3. 在 Coolify 建立新服務,設定環境變數和對外網址。
4. 接上公司 SSO(如果需要登入)。
5. 寫一份 **Runbook**(維運手冊):環境變數清單、部署步驟、出事怎麼回滾、負責人是誰。

> ✨ 這個清單就像「畢業檢查表」,確保小工具升級時不會臨時漏東漏西,結果上線出事。

## 五、環境變數怎麼管?Replit 和 Coolify 的做法對照

### 5.1 Replit 的做法

先澄清一個常見誤會:「Replit 設定都放在 `.replit` 裡」—— **錯!**
`.replit` 只負責「程式怎麼跑」,密碼和金鑰完全不放在那裡,而是走獨立的 Secrets 系統。

| 機制 | 做什麼 | 會進 git 嗎 | 什麼時候用 |
| --- | --- | --- | --- |
| **Secrets 面板** | 存密碼、API Key、連線字串 | ❌ 不會(加密存在 Replit 後端) | 放所有機密設定的主要地方 |
| **`.replit` 檔案** | 定義執行指令、入口點、套件 | ✅ 會 | **只放非機密**的結構設定 |
| **Account / Workspace Secrets** | 跨多個專案共用 | ❌ 不會 | 多個專案共用同一組金鑰時 |
| **Deployments Secrets** | 正式部署用的獨立環境變數 | ❌ 不會 | 開發和正式環境想用不同金鑰時 |

**程式怎麼讀**
- Node.js: `process.env.DATABASE_URL`
- Python: `os.environ['DATABASE_URL']`

### 5.2 Coolify 的做法

| 機制 | 做什麼 | 會進 git 嗎 | 什麼時候用 |
| --- | --- | --- | --- |
| **Environment Variables 面板** | 存密碼、API Key、連線字串 | ❌ 不會(加密存在 Coolify) | 放所有機密設定的主要地方 |
| **Build-time vs Runtime 變數** | 分成「打包時用」和「執行時用」 | ❌ 不會 | 前端打包需要的變數要設成 build-time |
| **Shared Variables** | 跨多個服務共用 | ❌ 不會 | 同一個專案的多個服務共用(如 DB 連線) |
| **`Dockerfile` / `docker-compose.yml`** | 定義執行環境、打包步驟 | ✅ 會 | 非機密的結構設定 |

**程式怎麼讀**
- 和 Replit **完全一樣**:`process.env.XXX` / `os.environ['XXX']`

### 5.3 兩邊直接比一比

| 項目 | Replit | Coolify |
| --- | --- | --- |
| 機密存在哪 | Secrets 面板 | Environment Variables 面板 |
| 非機密結構設定 | `.replit` | `Dockerfile` / `docker-compose.yml` |
| 開發 vs 正式環境怎麼分 | Secrets vs Deployment Secrets | 開不同 Service(staging / production) |
| 跨專案共用 | Account / Workspace Secrets | Shared Variables |
| 程式讀取方式 | `process.env` / `os.environ` | `process.env` / `os.environ` |
| 會不會進 git | Secrets 不會,`.replit` 會 | Env Vars 不會,Dockerfile 會 |

💡 **結論**:兩邊底層邏輯**完全一樣** —— 機密靠平台的 UI 管、程式只讀環境變數。
只要專案遵守「不寫死、不把密碼放進 git」,搬家就只是「換個 UI 點一點」的工夫而已。

## 六、環境變數的管理原則

每個正式專案都要有一份 `.env.example`,涵蓋以下類別(實際欄位由各專案設計階段決定):

- **基礎執行環境**:環境識別、服務埠、對外網址
- **資料來源**:資料庫、快取等連線設定
- **身份驗證**:SSO、JWT / Session
- **第三方服務**:郵件、金流、儲存、地圖、簡訊等
- **記錄與監控**:Log 等級、錯誤追蹤

**每個環境的設定位置**

| 環境 | 設定位置 |
| --- | --- |
| 本地開發 | 專案根目錄的 `.env` 檔(要加進 `.gitignore`) |
| Replit | Secrets 面板 |
| Coolify staging(測試站) | Service → Environment Variables(staging) |
| Coolify production(正式站) | Service → Environment Variables(production) |

**三條鐵則**
1. `.env.example` 只列**名稱**,絕對不寫真實值,而且**一定要**進 git。
2. 新增環境變數時,要**同時**更新 `.env.example` 和所有環境的設定,少一個都不行。
3. 程式啟動時自動比對 `.env.example` 檢查,少了哪個變數就立刻報錯停掉(fail fast)。
